# MCAP_CPP 开发记录（2026-06-09）

## 目标

将视频文件转换为 MCAP 文件，并使用 Foxglove 进行可视化验证。

整体流程：

```text
Video
 ↓
OpenCV读取视频
 ↓
JPEG压缩
 ↓
foxglove::CompressedImage
 ↓
Protobuf序列化
 ↓
MCAP写入
 ↓
Foxglove播放
```

---

# 一、CLI11 参数设计

## 需求

支持以下命令行参数：

| 参数 | 含义     |
| -- | ------ |
| -v | 视频路径   |
| -l | Log路径  |
| -t | Txt路径  |
| -o | 输出路径   |
| -c | 标定文件路径 |

其中：

```text
-l 和 -t 不会同时出现
```

---

## CLI11写法

```cpp
CLI::App app{"MCAP Converter"};

std::string videoPath;
std::string logPath;
std::string txtPath;
std::string outputPath;
std::string calibPath;

app.add_option("-v,--video", videoPath, "Video Path");
app.add_option("-l,--log", logPath, "Log Path");
app.add_option("-t,--txt", txtPath, "Txt Path");
app.add_option("-o,--output", outputPath, "Output Path");
app.add_option("-c,--calib", calibPath, "Calibration Path");

CLI11_PARSE(app, argc, argv);
```

---

## 参数顺序问题

CLI11 不关心参数顺序。

以下写法完全等价：

```bash
app.exe -v test.mp4 -o out
```

```bash
app.exe -o out -v test.mp4
```

```bash
app.exe -c calib.json -v test.mp4 -o out
```

# 二、OpenCV读取视频FPS

## 方法1

```cpp
double fps =
    cap.get(cv::CAP_PROP_FPS);
```

优点：

* 简单
* 快速

缺点：

* 依赖OpenCV后端
* 有时不准确

---

## 方法2

使用FFmpeg获取

```cpp
stream->avg_frame_rate
```

```cpp
double fps =
    av_q2d(stream->avg_frame_rate);
```

回退：

```cpp
stream->r_frame_rate
```

---

## 结论

推荐：

```cpp
avg_frame_rate
```

优先。

OpenCV内部很多情况下也是从FFmpeg获取。

但FFmpeg直接读取更可靠。

# 四、Protobuf Timestamp问题

## 错误

```cpp
payload.mutable_timestamp()->set_sec(0);
payload.mutable_timestamp()->set_nsec(0);
```

编译失败：

```text
No member named set_nsec
```

---

## 原因

Foxglove使用：

```proto
google.protobuf.Timestamp
```

字段为：

```proto
int64 seconds
int32 nanos
```

---

## 正确写法

```cpp
payload.mutable_timestamp()
    ->set_seconds(0);

payload.mutable_timestamp()
    ->set_nanos(0);
```

# 七、Schema构造问题

## 初始问题

Foxglove报错：

```text
no such type:
foxglove.CompressedImage
```

---

## 原因

Schema只有：

```cpp
schema.name
schema.encoding
```

没有：

```cpp
schema.data
```

---

## Foxglove要求

Schema必须包含：

```text
FileDescriptorSet
```

---

## 正确做法

```cpp
google::protobuf::FileDescriptorSet fds;
```

获取：

```cpp
foxglove::CompressedImage
    ::descriptor()
```

对应的：

```cpp
FileDescriptor
```

序列化为：

```cpp
schema.data
```

## `schema.id` 和 `channel.id` 可以一样吗?

**可以。**

MCAP里这两个 ID 属于不同命名空间。

例如：

```cpp
mcap::Schema schema;
schema.id = 1;

mcap::Channel channel;
channel.id = 1;
channel.schemaId = 1;
```

这是完全合法的。

MCAP 内部区分：

```text
Schema Table
------------
id=1 -> foxglove.CompressedImage

Channel Table
-------------
id=1 -> /compressed_image
schemaId=1
```

不会冲突。

---

## 多个 Channel 可以引用同一个 Schema 吗？

可以。

例如：

```cpp
Schema:
    id=1
    foxglove.CompressedImage

Channel:
    id=1
    /front_camera
    schemaId=1

Channel:
    id=2
    /rear_camera
    schemaId=1
```

这是标准用法。

---

## 多个 Schema 可以使用同一个 id 吗？

**绝对不行。**

例如：

```cpp
Schema id=1
CompressedImage

Schema id=1
RawImage
```

这是非法的。

---

## 多个 Channel 可以使用同一个 id 吗？

**绝对不行。**

例如：

```cpp
Channel id=1
/front

Channel id=1
/rear
```

这是非法的。

---

正常 MCAP：

```text
Header

Schema

Channel

Message

Message

Message

Chunk

DataEnd

Footer
```

# 十、Chunk start offset问题

## 错误

```text
Chunk start offset 46
does not point to chunk record
(found DataEnd)
```

---

## 初步怀疑

怀疑：

```cpp
closeLastChunk()
```

或者：

```cpp
Summary
Footer
```

写入错误。

---

## 实际根因

最终排查发现：

```cpp
McapWriter
```

设计成了：

```cpp
Singleton
```

---

### 原设计

```cpp
McapWriter::Instance()
```

全局单例。

多个模块：

```cpp
Schema
Channel
Writer
```

共享同一个实例。

---

### 结果

生命周期顺序异常：

```text
Schema析构
 ↓
Channel析构
 ↓
Writer关闭
 ↓
Footer损坏
 ↓
ChunkIndex错误
```

Foxglove报：

```text
Chunk start offset 46
```

---

## 修复

取消单例设计。

改为：

```cpp
class ConvertToMcapData
{
    McapWriter writer_;
};
```

统一生命周期管理。

---

## 结果

问题彻底解决。

Foxglove正常读取MCAP。

# 十一、MCAP结构理解

## Schema

定义消息类型：

```cpp
mcap::Schema
```

例如：

```text
foxglove.CompressedImage
```

---

## Channel

定义Topic：

```cpp
mcap::Channel
```

例如：

```text
/compressed_image
```

---

## Message

具体消息：

```cpp
mcap::Message
```

包含：

```cpp
channelId
logTime
publishTime
data
```

---

## 对应关系

```text
Schema
    ↓
Channel
    ↓
Message
```

# 今日结论

本次开发过程中最大的坑并不是：

* OpenCV
* Protobuf
* Foxglove

而是：

```text
MCAP Writer生命周期管理
```

最终问题根因：

```text
McapWriter使用Singleton设计
导致关闭顺序异常
破坏ChunkIndex和Footer
最终Foxglove无法正确读取MCAP
```

修复后：

```text
视频 -> JPEG -> CompressedImage -> MCAP
完整链路打通。
```
