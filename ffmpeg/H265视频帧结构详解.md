# H.265 / HEVC 的帧结构详解 🎬

H.265（HEVC, High Efficiency Video Coding）相比 H.264 最大的变化之一，就是：

* 编码块结构更复杂
* 帧间预测更灵活
* 压缩率更高（同画质下码率约省 30%~50%）

你问的“帧结构”，其实可以从几个层次理解：

---

# 一、HEVC 数据层级结构（从大到小）

HEVC 的视频流结构：

```text
Video
 └── Sequence（序列）
      └── GOP（Group of Pictures）
           └── Frame（帧）
                └── Slice（切片）
                     └── CTU（Coding Tree Unit）
                          └── CU（Coding Unit）
                               └── PU（Prediction Unit）
                               └── TU（Transform Unit）
```

这是理解 HEVC 的核心。

---

# 二、GOP（图像组）结构

GOP 是帧组织方式。

HEVC 仍然有：

* I 帧
* P 帧
* B 帧

但 B 帧能力远强于 H264。

---

# 三、帧类型详解

---

## 1. I 帧（Intra Frame）

特点：

* 不依赖其它帧
* 独立解码
* 类似 JPG 图片

作用：

* 随机访问
* 视频起点
* 场景切换

HEVC 中：

* 使用帧内预测（Intra Prediction）
* 支持 35 种预测方向（H264 只有 9 种左右）

例如：

```text
I帧
 ├── 当前块像素
 └── 从本帧邻近块预测
```

---

## 2. P 帧（Predictive Frame）

特点：

* 参考前面的帧
* 使用运动估计

原理：

```text
当前块 ≈ 前一帧某块 + 残差
```

编码内容：

* motion vector（运动向量）
* residual（残差）

---

## 3. B 帧（Bi-predictive Frame）

HEVC 的核心强化点之一 🚀

可以：

* 参考前帧
* 参考后帧
* 多参考帧
* 双向预测

例如：

```text
B帧 = 前帧预测 + 后帧预测 + 残差
```

HEVC 的 B 帧甚至可以：

* 被其它 B 帧引用
* 形成层级 B（Hierarchical B）

---

# 四、HEVC 的真正核心：CTU 树结构 🌲

H264：

```text
Macroblock = 16x16
```

HEVC：

```text
CTU 最大可达 64x64
```

这是压缩率提升的关键。

---

# 五、CTU（Coding Tree Unit）

HEVC 基本编码单元。

例如：

```text
CTU 64x64
```

然后：

```text
64x64
 ├── 32x32
 │    ├── 16x16
 │    └── 16x16
 └── 32x32
```

像四叉树一样递归分裂。

这叫：

```text
Quad Tree Partition
```

---

# 六、CU（Coding Unit）

CU 决定：

```text
这个区域怎么编码？
```

可能：

* Intra（帧内）
* Inter（帧间）

比如：

```text
CU
 ├── 编码模式
 ├── QP
 └── 分裂信息
```

---

# 七、PU（Prediction Unit）

PU 决定：

```text
如何预测？
```

例如：

```text
PU
 ├── Motion Vector
 ├── Reference Frame
 └── Prediction Mode
```

---

# 八、TU（Transform Unit）

TU 负责：

```text
残差变换
```

即：

```text
Residual
   ↓ DCT/DST
Frequency Domain
   ↓ Quantization
Compressed Data
```

---

# 九、HEVC 的 Slice 结构

一帧可以分成多个 Slice：

```text
Frame
 ├── Slice 1
 ├── Slice 2
 └── Slice 3
```

作用：

* 并行解码
* 错误恢复
* 网络传输

---

# 十、NALU 结构（重点）🔥

HEVC 数据最终通过：

```text
NAL Unit
```

传输。

类似 H264，但 Header 不同。

---

## NALU 基本结构

```text
+---------------+
| Start Code    |
+---------------+
| NAL Header    |
+---------------+
| RBSP Payload  |
+---------------+
```

---

## Start Code

通常：

```text
00 00 01
```

或者：

```text
00 00 00 01
```

用于分隔 NALU。

---

# 十一、HEVC NAL Header

HEVC 的 NAL Header：

```text
forbidden_zero_bit    1 bit
nal_unit_type         6 bits
nuh_layer_id          6 bits
nuh_temporal_id_plus1 3 bits
```

比 H264 更复杂。

---

# 十二、NALU 类型

常见：

| 类型    | 说明                     |
| ----- | ---------------------- |
| VPS   | Video Parameter Set    |
| SPS   | Sequence Parameter Set |
| PPS   | Picture Parameter Set  |
| IDR   | 关键帧                    |
| TRAIL | 普通帧                    |
| SEI   | 附加信息                   |

---

# 十三、VPS / SPS / PPS

---

## VPS

HEVC 新增。

管理：

* 编码层级
* profile
* level
* 多层视频

---

## SPS

包含：

* 分辨率
* bit depth
* chroma format
* CTU size

---

## PPS

包含：

* slice 参数
* entropy coding 参数
* QP

---

# 十四、IDR 帧

HEVC 的真正关键帧。

特点：

```text
后续帧不能引用 IDR 前面的帧
```

因此：

* 可随机 seek
* 可切流

---

# 十五、CRA / BLA（HEVC 新增）

HEVC 比 H264 多了更多随机访问帧。

例如：

| 类型  | 含义                 |
| --- | ------------------ |
| IDR | 完全随机访问             |
| CRA | 清洁随机访问             |
| BLA | Broken Link Access |

用于：

* 流媒体
* DASH
* HLS

---

# 十六、参考帧结构

HEVC 支持：

* 更多参考帧
* 长期参考帧
* 多层 B 帧

例如：

```text
I B B P B B P
```

实际上可能：

```text
display order:
0 1 2 3 4 5 6

decode order:
0 3 1 2 6 4 5
```

因为 B 帧需要未来帧。

---

# 十七、POC（Picture Order Count）

HEVC 中非常重要。

用于：

```text
显示顺序 ≠ 解码顺序
```

播放器靠 POC 重排。

---

# 十八、CABAC 熵编码

HEVC 强制使用：

```text
CABAC
```

不再支持 CAVLC。

压缩率更高，但编码更复杂。

---

# 十九、Tiles / WPP（并行编码）

HEVC 增强了并行能力。

---

## Tiles

把画面切块：

```text
+----+----+
| T1 | T2 |
+----+----+
| T3 | T4 |
+----+----+
```

不同线程编码。

---

## WPP（Wavefront Parallel Processing）

逐行波前并行：

```text
Row1 →
  Row2 →
    Row3 →
```

现代硬件编码大量使用。

---

# 二十、一个完整 HEVC 码流例子

```text
[VPS]
[SPS]
[PPS]
[IDR Frame]

[P Frame]
[B Frame]
[B Frame]
[P Frame]
...
```

而每帧内部：

```text
Frame
 └── Slice
      └── CTU
           └── CU
                ├── PU
                └── TU
```

---

# 二十一、HEVC vs H264 核心区别

| 特性      | H264        | H265       |
| ------- | ----------- | ---------- |
| 最大块     | 16x16 MB    | 64x64 CTU  |
| 分割方式    | 较简单         | 四叉树        |
| B帧      | 普通          | 层级B        |
| Intra方向 | 9           | 35         |
| 熵编码     | CABAC/CAVLC | CABAC only |
| 并行      | 一般          | Tiles/WPP  |
| 压缩率     | 基准          | 提升30~50%   |

---

# 二十二、解码器真正做的事

解码流程：

```text
NALU解析
 → SPS/PPS
 → Slice解析
 → 熵解码(CABAC)
 → 反量化
 → 反变换
 → 运动补偿
 → 帧重建
 → 去块滤波
 → SAO
 → 输出YUV
```

---

# 二十三、HEVC 的后处理

---

## Deblocking Filter

消除块边缘。

---

## SAO（Sample Adaptive Offset）

HEVC 新增。

进一步修复：

* ringing
* banding

画质提升明显。

---

# 二十四、实际工程里最重要的几个东西 ⚙️

做播放器/编解码/流媒体时：

重点通常是：

---

## 1. NALU 提取

AnnexB：

```text
00 00 01
```

AVCC/HVCC：

长度前缀。

---

## 2. VPS/SPS/PPS

必须先收到。

否则无法解码。

---

## 3. POC 重排

显示顺序问题。

---

## 4. DPB（Decoded Picture Buffer）

保存参考帧。

B帧大量依赖。

---

# 二十五、HEVC AnnexB 示例

```hex
00 00 00 01 40 ... VPS
00 00 00 01 42 ... SPS
00 00 00 01 44 ... PPS
00 00 00 01 26 ... IDR
```

---

# 二十六、为什么 H265 编码慢 🐢

因为：

* 四叉树搜索
* 更多预测模式
* 更多运动搜索
* 更多参考帧
* CABAC 更复杂

本质：

```text
用大量计算换码率
```

---

# 二十七、最核心的一句话总结 🎯

HEVC 的本质：

```text
更大的块
+
更灵活的分裂
+
更复杂的预测
+
更强的熵编码
```

从而实现：

```text
更低码率、更高画质
```
